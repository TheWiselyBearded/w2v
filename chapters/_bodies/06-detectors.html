Part III В· Chapter 6

The Detectors вҖ” the swappable part.

This is the first chapter about the model itself, and its main job is to convince you that the model matters less than you think. There are two great families of object detector, you almost never train either from scratch, and the biggest lever on accuracy is not the architecture at all вҖ” it is the data. The detector is a component you slot into the pipeline, not the pipeline itself.

Five chapters in, we have built a system that can manufacture labeled skies, translate every source into one canonical record, and freeze datasets into reproducible snapshots. We have not yet trained a single model. That ordering was deliberate. The recurring thesis of this book вҖ” the pipeline is the product вҖ” is easiest to believe once you see how interchangeable the detector really is. In this chapter we plug one in, look honestly at what changes when we swap it for another, and discover that the answer is: less than the marketing would have you expect.

6.1 What an object detector actually does

Strip away the vocabulary and a detector is a function with a very simple signature. It takes one image and returns a set of guesses. Object detection is the task of finding every object of interest in an image and, for each one, producing three things: a bounding box (the rectangle in pixel coordinates that tightly encloses the object), a class (which category it belongs to, drawn from your fixed taxonomy вҖ” "aircraft", "bird", "drone", or simply "object"), and a confidence, sometimes called the score (a number from 0 to 1 saying how sure the model is that this box contains a real object of that class). That is the entire output: a list of (box, class, confidence) triples, and nothing more. It matches the detection fields of the canonical record from Chapter 2 exactly вҖ” by design, because the detector's output is a canonical record.

Every detector ever built produces that same list. What differs вҖ” wildly вҖ” is how it arrives at it. There are two dominant strategies, and understanding the split is enough to make sense of the entire zoo of model names you will encounter.

6.2 Family one: tile the image and vote (anchor / grid)

The first family thinks spatially. Imagine laying a grid over the image and, at every cell, asking a fixed menu of questions: is there an object centered near here? If so, how much bigger or smaller than this reference box is it, and what class is it? The reference boxes are called anchors вҖ” candidate box shapes and scales tiled densely across the image; newer "anchor-free" variants drop the hand-drawn shapes and predict directly from grid points, but the spirit is the same. The model looks everywhere at once and emits a prediction for every tile.

This produces a flood of overlapping boxes вҖ” a single object lights up several neighboring cells. To clean it up, the family uses Non-Maximum Suppression (NMS): a post-processing step that walks the boxes from highest confidence down, keeps each one, and deletes any lower-scoring box that overlaps it too much. One object, one surviving box. This lineage is the YOLO family (You Only Look Once); a clean, permissively licensed representative is YOLOX. Its defining virtues are speed and predictability. It is the iteration workhorse вҖ” the detector you reach for when you want to run an experiment before lunch and read the result after.

6.3 Family two: a fixed set of questions (query-based / DETR)

The second family thinks differently. Instead of tiling space, it carries a fixed set of object queries вҖ” say 300 learned "slots," each of which independently asks "is there an object, and if so, where and what?" This is the DETR architecture (DEtection TRansformer); a permissively licensed representative is RF-DETR. The queries are not tied to grid positions; they are learned probes that attend over the whole image.

The elegant part is how it is trained. DETR does set prediction вҖ” it outputs a set of boxes all at once and is scored on the set as a whole. During training, each ground-truth object is assigned to exactly one query using bipartite matching (the Hungarian matching algorithm finds the one-to-one pairing of predictions to truths that minimizes total error). Because every real object is matched to precisely one query, there are no duplicate boxes to remove вҖ” DETR needs no anchors and no NMS at all. The suppression step that the grid family cannot live without simply does not exist here. The price is training cost: query models are heavier and slower to train, and they typically need more data and more epochs before they come good. The reward is accuracy вҖ” when they converge, they tend to edge out the grid family, especially on cluttered scenes.

A DETR model always emits its full complement of predictions вҖ” 300 boxes, most of them the model's way of saying "nothing here." A tempting but broken metric is cardinality: compare how many objects the model predicted against how many exist. Because the raw prediction count is structurally pinned near the query count no matter what is in the frame, this number tells you almost nothing about quality вҖ” a perfect model and a useless one can post identical cardinality. Do not use it. Judge the model with precision, recall, and F1 measured at swept confidence thresholds, which we build in Chapter 9.

6.4 You do not start from scratch: transfer learning

Here is a fact that surprises newcomers and quietly defines the economics of the whole field: you almost never train a detector from nothing. Instead you practice transfer learning вҖ” you take a model that already learned to see on a large, general dataset and adapt it to your problem. The reusable core is the backbone: the stack of early layers that turn raw pixels into general visual features like edges, corners, and textures. That backbone arrives as pretrained weights вҖ” numbers already trained, typically on a big public dataset such as COCO, so the model starts knowing how to look at images in general. You then perform fine-tuning: continuing training on your own, much smaller dataset so the model re-specializes to your objects and your domain.

Why does this work? Because the low-level features are almost universal. The filters that detect an edge against the sky are the same filters that detect an edge against a street; a network does not need to relearn what a gradient is just because the subject changed. Most of what fine-tuning teaches is concentrated in the head вҖ” the final layers that decide "this feature pattern means aircraft." You are not rebuilding vision; you are re-teaching the last mile. This is why a few thousand well-chosen frames can produce a usable detector, and it is the bridge that makes the synthetic-data strategy of Part II economically sane.

6.5 Licensing is an engineering gate, not a footnote

Transfer learning means you are shipping other people's numbers, and that makes licensing a first-class engineering concern rather than a legal afterthought. The single most important thing to internalize is that three separate licenses are in play and they are not the same license: the license on the code, the license on the pretrained weights, and the license on the training data those weights were derived from.

The traps are real and specific. Some detector families release small model sizes under a permissive license such as Apache-2.0 but place their larger, higher-accuracy variants under a restrictive or commercial license вҖ” so the exact checkpoint you download changes your rights. Some of the most popular YOLO implementations ship under GPL/AGPL terms that require explicit organizational clearance for many deployment scenarios. And pretrained weights routinely carry terms the code repository does not, because the weights inherit obligations from the data they were trained on. You cannot infer any of this from the repo's headline license.

вң— The rule

Verify the exact license for the exact code, the exact weights file, and the training data you actually use вҖ” every one, separately. Never infer weight rights from a repository's code license. Then encode "weights license approved" as a promotion gate, exactly like the fail-closed license gate on datasets in Chapter 5. A model whose weights have not cleared review does not get promoted, no matter how well it scores. The gate fails closed: unknown provenance is treated as forbidden until proven otherwise.

6.6 Feeding the trainer: export, track, and name the job

Training frameworks do not read our canonical record directly; they expect a standard on-disk layout вҖ” a COCO-format JSON of annotations beside train/, valid/, and test/ image folders. So the first step of any training run is to deterministically export an immutable dataset snapshot (Chapter 5) into that layout. Deterministic matters: the same snapshot must always produce byte-identical folders, or reproducibility dies at the first step.

Every run is then recorded by an experiment tracker вҖ” the examples in this book use MLflow вҖ” which captures the config, the metrics over time, and the output artifacts, so any two runs can be compared honestly instead of from memory. And every training job is given a reproducible identity: a name plus a config_hash(experiment + detector + dataset id). Re-submitting the same identity does not silently spawn a new thing; it creates attempt N under the same job. Reproducibility becomes a hashable property you can point at.

в–ё Epochs and config are part of the hash

The hash includes the training configuration вҖ” epoch count, learning rate, augmentation settings, everything. Change the epochs and you have not resumed the old job; you have defined a new job with a new identity. This is a feature: it means "the same job" always means the same recipe, so two attempts under one job differ only by the luck of the hardware, never by a quietly edited knob.

6.7 Reproducibility is genuinely hard вҖ” a war story

You would hope that two attempts of the same hashed job produce the same model. They do not, quite, and pretending otherwise will burn you. The culprit is nondeterministic GPU kernels: some fast CUDA/cuDNN operations accumulate results in a nondeterministic order, so identical code on identical data drifts apart run to run.

The last-step weights that lied

compare the checkpoint you ship, not the last step

In one build, a team ran the same job twice and compared the models by their raw final-step weights вҖ” the parameters at the literal last training iteration вҖ” on a small validation set. The two attempts disagreed violently: one posted an mAP near 0.22, the other near 0.90, from the same recipe. Panic. Was the pipeline broken?

It was not. The raw last-step weights are the noisiest thing in the whole system вҖ” a single unlucky final batch can throw them anywhere. But nobody ships the last-step weights. What you actually promote is the EMA checkpoint (exponential moving average вҖ” a smoothed running average of the weights across training that filters out per-step jitter). Compared on the EMA weights, the same two attempts were stable and close: about 0.64 and 0.74. The disagreement lived entirely in the noisy artifact nobody uses.

Three lessons came out of it, and all three are now standing policy. First, compare the checkpoint you will actually ship вҖ” the EMA or best-by-metric weights вҖ” never the raw last step. Second, pin deterministic GPU settings (the cuDNN deterministic flags) to shrink run-to-run drift wherever the speed cost is affordable. Third, a tiny validation set makes every metric quantized and jumpy: when there are only a handful of objects, one box flipping right or wrong swings the whole number. Grow the validation set вҖ” a theme we return to in Chapter 9.

6.8 The architecture verdict: data dominates

So which family wins? The honest answer is the one that vindicates the whole book. At small data scale the two architectures can look far apart, and it is tempting to declare a champion. But as the data grows and diversifies, they converge.

At pilot scale the DETR model led by a wide margin. At full data scale the same two architectures finished at roughly 0.857 and 0.841 AP@50 вҖ” a rounding error apart. The early gap was not really a fact about the architectures; it was a fact about how much data each needed to reach its own ceiling. Give both enough, and the ceilings are almost the same height.

That converts the choice into a practical one about your workflow, not a search for the "best" model. Pick an RF-DETR-style query model when you want peak accuracy and can afford the training bill. Pick a YOLOX-style grid model when you want to iterate fast вҖ” it can run several times more epochs per hour, and in a data-centric loop the team that iterates faster usually wins. Either way, the biggest lever on your accuracy is not on this page. It is the data: its scale, its diversity, and how faithfully it matches the real sky. That is precisely why Parts II and IV вҖ” building better data and closing the loop on failures вҖ” dominate this book, and this chapter is comparatively short.

The detector is the most swappable component in the system. Choose it for how it fits your iteration loop, not for a leaderboard вҖ” the data decides the ceiling.

6.9 One more swappable piece: the tracker

A detector sees each frame in isolation. Turning a stream of per-frame boxes into stable object identities over time is the job of a tracker, and it too is a plugin behind a common interface, not a fixed part of the model. Start with a deterministic motion-and-association tracker вҖ” predict where each box should go and match it to the nearest new detection вҖ” and add appearance embeddings only if a benchmark proves you need them. The full treatment waits for Chapter 11; for now, note only that the same discipline applies: keep it swappable, and let evidence, not fashion, decide when to make it fancier.

Takeaways
  • A detector is a simple function. Image in, a list of (box, class, confidence) out вҖ” which is exactly the canonical detection record. Everything else is implementation.
  • Two families, one output. Grid detectors (YOLOX) tile densely and dedupe with NMS вҖ” fast and predictable; query detectors (RF-DETR) do set prediction with bipartite matching вҖ” no NMS, more accurate, heavier to train.
  • Transfer learning is the default. Start from a pretrained backbone and fine-tune; you re-teach the head your objects, you do not rebuild vision.
  • Licensing is a promotion gate. Code, weights, and data carry separate licenses вҖ” verify all three for the exact artifacts, fail closed, and never infer weight rights from code.
  • Reproducibility is hard; compare the EMA checkpoint you ship, not the noisy last step, pin deterministic GPU settings, and grow tiny validation sets.
  • Data dominates architecture. Two models converge as data scales (0.857 vs 0.841 at full scale) вҖ” so the detector is swappable, and the data is the product.
