Part II · Chapter 3

Building Worlds — manufacturing skies a computer already understands.

Real footage of small aerial objects is scarce, expensive, and legally encumbered — so we render our own. This chapter is about generating the worlds: how to choose a simulator, why byte-for-byte determinism is the whole discipline, and how to sample scenarios that look like the hard cases from Chapter 1 rather than the easy ones.

Chapter 1 ended on a wall: the labeled data you need barely exists, and the labels are the real cost — a human boxing twenty-pixel specks by the million is an industrial undertaking. This chapter walks through the door on the far side of that wall. If you cannot buy enough labeled skies, you build them. And when you build a sky in a simulator, something wonderful happens to the labeling problem: it disappears. The computer placed every object in the scene, so it already knows each object's exact position, size, and identity. There is nothing to label because nothing was ever unknown. The labels are free, and — unlike a tired human at 2 a.m. squinting at a five-pixel smudge — they are perfect.

That is the promise. The catch, which we will spend all of Part III paying down, is that a rendered sky is not a photographed one. But before we can worry about realism, we have to be able to generate worlds at all, at scale, reproducibly. This chapter is about the machine that makes worlds; Chapter 4 is about how it turns those worlds into the perfect boxes just promised.

3.1 Choosing a simulator

The first real decision is which engine renders your worlds, and there is no universally correct answer — only trade-offs you should make on purpose. A simulator here is just a piece of software that renders images of a 3D scene you describe; the candidates fall into three broad families.

EngineStrengthCost you pay
Game engine (e.g. Unity) Fastest to start; excellent hand control over custom scenes; huge asset ecosystem. You write more of your own annotation and export code — the engine does not hand you synthetic labels.
NVIDIA Omniverse Replicator Strongest programmatic domain randomization; OpenUSD scene composition; headless generation; synthetic-data workflows built in. Heavier stack and a real asset/USD learning curve before the first image comes out.
Unreal Superb rendering and a deep environment/landscape ecosystem. More custom integration to get annotation and physical scale wired up the way you need.

The pragmatic recommendation: start with one engine, and keep a second as a second renderer for later. You want one path working end-to-end before you spread yourself across two. But you also want the option of a second renderer eventually, because agreement between two independent engines is one of the few cheap defenses against a whole class of rendering bugs.

Here is the architectural point that makes "keep a second renderer" painless rather than terrifying. Recall the narrow waist from Chapter 2: every source is translated into one canonical record at the door. Your simulator is just another source. The exporter — the code that takes a rendered scene and emits data — emits the canonical record, regardless of which engine drew the pixels. Training never reads Unity's native format or Omniverse's; it reads the one record. That single commitment means adding a second engine is writing one more adapter, not rebuilding the pipeline.

◇ Exporter, not "the sim"

We are careful to separate the engine (which renders pixels) from the exporter (which decides scene contents, steps the world, and writes canonical records). The engine is replaceable. The exporter is where your discipline lives — the RNG, the IDs, the frame loop, the provenance. When this chapter says "own your generator," it means the exporter.

3.2 Determinism is the whole discipline

If you remember one thing from this chapter, remember this: a synthetic-data generator that is not reproducible is barely worth building. The entire value of "the computer knows the answer" rests on being able to ask the same question twice and get the same world back. We call this determinism: same seed plus same generator version produces identical output bytes, every time, on every machine.

Why do you care about identical bytes, not just "similar-looking runs"? Because Chapter 5 identifies datasets by content hash — a dataset's name is a fingerprint of its exact contents. If regenerating the same scenario produces even slightly different pixels, the hash changes, the lineage breaks, and you lose the ability to regenerate exactly what produced a published result. Determinism is what makes a dataset a citable fact rather than a fading memory. Three requirements get you there.

1 · Own your random numbers

Do not draw from the engine's shared random-number generator. A game engine's RNG is a global resource that particle effects, animation, and a hundred other subsystems also pull from — so the draws your scene depends on shift the instant anything else in the frame changes. Instead, carry your own pseudo-random number generator (a deterministic algorithm that turns a seed into a repeatable stream of numbers), seeded per run. A common, robust choice is to use SplitMix64 to expand your seed into the state of an xorshift128+ stream, and draw every sampled value from that stream alone. Nothing else in the process can perturb it.

2 · Content-derived stable IDs

Every object needs an identity that is the same on every run — the object_id the canonical record demands. Do not use a random or auto-incrementing counter, because those drift. Instead derive a deterministic UUID (an RFC-4122 version 5 identifier, which is a hash of some input string rather than a random draw) from the tuple (generator version, config name, seed, object index). The same object in the same scenario gets the same id forever — which is exactly what makes the data sequence-aware across regenerations, not just within a single run.

3 · A synchronous frame loop — no physics engine

This is the one people get wrong, and §3.5 is a whole war story about it. You must step object motion yourself, in a plain synchronous loop with a fixed time step. No physics engine. No coroutines, no async, no reliance on the engine's variable frame timing. Engine physics resolves collisions with tiny nondeterministic differences, and real frame delta-times wobble under load — either one silently corrupts reproducibility. You advance each object by velocity × dt with a dt you chose, and you render. That is all.

Same seed + same generator version = identical bytes. Anything that breaks that equality is a bug, no matter how good the pictures look.

3.3 Domain randomization: a scenario is a distribution

A single rendered sky teaches a model almost nothing — it will memorize that one sky. The power of synthetic data comes from domain randomization: instead of authoring one scene, you define a scenario as a distribution over scenes and sample it, recording every sampled value and the seed that produced it. Train across ten thousand samples from a wide distribution and the model is forced to learn what is invariant — the object — rather than the accidental backdrop.

The axes you randomize are, in effect, a checklist of everything Chapter 1 warned you about:

  SCENARIO = a distribution you sample, logging every draw + the seed
  ────────────────────────────────────────────────────────────────────
  object     ▸ type · appearance · orientation
  SIZE       ▸ pixel extent — LOG-UNIFORM, far-biased  ◄── the hard regime
  scene      ▸ background · geography · terrain · clutter
  weather    ▸ cloud · haze · sun angle · contrast
  camera     ▸ pose · exposure · gain · sensor noise
  optics     ▸ lens effects · compression artifacts · motion blur
  motion     ▸ relative velocity · occlusion · approach dynamics (§3.4)
  negatives  ▸ EMPTY sequences — no object at all

Two of these deserve emphasis. First, pixel size: if you sample it uniformly you will fill your dataset with easy, large targets and starve the model of the hard small ones. Sample it log-uniform and far-biased so the distribution leans toward small and distant, matching the regime where the problem actually lives. Second, and easy to forget: deliberately include negative sequences — clips with no object present at all. A detector that has never seen empty sky does not know what "nothing" looks like, and a monitoring system that invents ghosts over blank sky is the false-alarm failure from Chapter 1 made manifest.

▸ Record the draw, not just the result

Every sampled value — every size, angle, exposure, and the seed behind them — goes into the scenario parameters written beside the sequence. This is what lets you later ask "which slice of the distribution does the detector fail on?" and answer it, instead of guessing. The sampled parameters are as much a part of the data as the pixels.

3.4 Approach dynamics and the terrain you forgot

Static randomization gets you variety; it does not get you sequences that behave like real approaches. Two additions close that gap.

Give the camera platform its own motion. The ownship — your own flying camera — flies level along its heading at a sampled speed, so the background sweeps the way it does from a real moving platform, killing the background-subtraction shortcut exactly as Chapter 1 described. Then give some targets an approach probability: with some chance, a target aims its velocity straight at the camera — a collision course — so that over the sequence it grows in the frame, three pixels now and forty pixels ten seconds from now. That growth is the size-change challenge from Chapter 1, reproduced on demand. You record own_speed and the count of approaching targets as scenario parameters, so the difficulty is a logged, tunable quantity rather than an accident.

One subtlety this creates: an approaching target on a near-miss will eventually overfly the camera and end up behind it. The labeler (Chapter 4) simply drops targets that are behind the camera, so a near-miss degrades cleanly to "no visible object" rather than producing a box for something outside the field of view.

⚠ The empty-sky mistake

The most common early error in this whole enterprise is generating targets against empty sky only. It renders fast, it looks clean, and it quietly builds a model that has never seen the ground. Real footage — especially the moment the camera tilts toward the horizon — is full of terrain: ridgelines, buildings, glinting water, and ten thousand high-contrast specks that each look like a small object. A model trained on blank sky treats every one of them as a target. You must deliberately synthesize terrain and clutter scenes. Chapter 7 quantifies exactly what it costs when you don't — it is the single largest sim-to-real gap most teams discover, and always the hard way.

3.5 War story: the seed that wasn't

Two identical seeds, two different datasets

What broke: reproducibility

An early exporter tied object motion to the game engine's built-in physics and coroutine timing. It looked reasonable — let the engine move things, let physics resolve the geometry, read the positions back out each frame. Objects flew, boxes appeared, the images looked great. Everyone moved on.

Then someone regenerated a scenario to reproduce a result and the dataset's content hash came out different — same seed, same config, different bytes. Not wildly different; a handful of objects sat a fraction of a pixel off. But a fraction of a pixel is a different hash, and a different hash is a different dataset as far as lineage is concerned. The cause was twofold: the physics solver resolved motion with tiny nondeterministic differences between runs, and the engine's frame delta-time varied with system load, so dt was never quite the same twice. Two runs that were supposed to be identical never were.

The fix was to rip engine physics out entirely: step motion synchronously in a plain loop with a fixed dt, seed a self-owned RNG (§3.2), and let the engine do nothing but render the positions the exporter dictated. After that, the same seed and version produced the same bytes, run after run, machine after machine. The lesson generalizes past this one bug: every source of nondeterminism you do not own is a source of nondeterminism you cannot control.

3.6 Own your assets, and run headless

Two operational commitments turn a clever renderer into a data factory the rest of the pipeline can trust.

Own every asset. If all your content is either procedurally generated or drawn from sources you can freely use, then the license-and-provenance gate from Chapter 5 can cleanly approve every single sequence. That is not a legal footnote — it is what lets the whole dataset ship. Freely reusable, permissively-licensed sources give you plenty of realism without entanglement: public-domain and CC0 model scans and textures (for example Smithsonian Open Access CC0, NASA's public-domain 3D models, and CC0 texture and HDRI libraries). Attach a license/provenance sidecar — a small companion file recording where every asset came from and under what license — to each sequence at generation time, not retroactively, so the record is never in doubt.

Run headless. The exporter must run without a display — in a batch job, a CI pipeline, or a cloud instance with no screen attached — driven by a scenario manifest (a file describing what to generate). Headless generation is what lets you produce data at scale instead of babysitting a GUI. It is also what makes generation a step the offline plane can automate, the same way it automates training and evaluation.

3.7 Build the blockout, then dress the set

Everything so far treats the engine's pixels as the finished product. They are not — and the reason previews Part III's central drama. A game engine is superb at structure: geometry, motion, occlusion, and the perfect labels that fall out of them. It is mediocre at surface statistics — the grain of real terrain, atmospheric scattering, the particular character of a real sensor. And it is exactly those surface statistics, especially in the background, that decide whether a model trained on rendered skies survives contact with photographed ones.

So borrow a trick from film production. A film set is built twice: first the blockout — plain geometry that fixes where everything is and how it moves — and then the set dressing that makes it look real. The engine renders the blockout. Then a second stage runs a generative image-to-image pass over each rendered frame: a diffusion model, conditioned on the rendered image at a deliberately low edit strength, repaints textures and atmosphere while leaving the scene's layout pinned. The geometry the labels were computed from does not move; the surfaces stop looking like a video game.

Left alone, a generative model will happily hallucinate — and a hallucinated pixel inside a bounding box is a corrupted label. Two commitments make the pass label-safe by construction:

  • Low edit strength. The denoising strength stays low (this build uses 0.35), so the model is only allowed to repaint surface statistics, not invent or move objects. Realism comes from the background, which is also where the false-positive gap lives.
  • Target preservation. After enhancement, the original rendered pixels inside every annotation box — padded slightly, with a feathered edge — are composited back over the enhanced frame. The pixels under each label are exactly the pixels the label was computed from, unconditionally. The generative model never gets a vote on the object.

And the discipline of this chapter does not relax just because a neural network joined the render path. The per-frame seed is derived by hashing the source frame's content with a fixed salt, so the same input sequence and parameters reproduce the same edits. The enhanced output is written as a new derived sequence with its own deterministic identity — never overwriting its source — carrying a provenance block that records the model, parameters, and the lineage back to the raw render. It then enters the pipeline through the same door as any other sequence: the canonical-record adapter. Training never learns that a diffusion model touched the pixels, the same way it never learns which engine drew them.

⚠ The generative model is an asset with a license

The enhancement model's weights are now part of your data's provenance, exactly like a texture or a 3D scan — so they go into the license sidecar and through the Chapter 5 gate. Choose weights under a genuinely permissive license: several popular image models ship under use-restricted licenses that a fail-closed license gate will (correctly) reject. And treat API-hosted image models — which offer no edit-strength control and no seed — as a sketchpad for eyeballing whether the idea works, not as a production path. The production path is a local model with a strength knob, a seed, and a license you have read.

3.8 What "done" looks like

How do you know the world-builder is good enough? Not by looking at the images and admiring them. Prettiness is a trap — a gorgeous renderer that does not transfer to real footage is worthless, and a plain one that does is priceless. The acceptance criteria are concrete:

  • One logical scenario manifest maps to every backend, with any engine-specific unsupported fields explicitly documented rather than silently ignored.
  • Generation is deterministic per seed and version — the byte-reproducibility of §3.2, verified, not assumed.
  • Generated boxes agree with an independent geometric check — the exporter's labels and a separate geometry computation must match (Chapter 4).
  • Every sequence carries its asset licenses and generator provenance, so the Chapter 5 gate can approve it without a human hunting down sources.
  • A bake-off measures throughput and transfer to real data — generation speed and how well a model trained on the output performs on real footage — never visual beauty.

Notice what every one of these criteria has in common: none of them is about the renderer being impressive. They are about the generator being trustworthy — reproducible, verifiable, licensed, and measured by the only thing that matters, which is whether the worlds it builds teach a model to see the real one. The pipeline is the product, and the world-builder is a pipeline stage like any other. It earns its place by feeding clean, canonical, content-addressed records into the same machine everything else reads from.

We now have worlds. In the next chapter we make them confess their contents: how the exporter turns a scene it fully understands into the perfect, geometry-derived boxes that make synthetic data worth the trouble.

Takeaways
  • Synthetic labels are free and perfect because the computer placed every object — the labeling problem from Chapter 1 evaporates when you generate the world.
  • Pick one engine, keep a second renderer for later, and make the exporter emit the canonical record so training never learns which engine drew the pixels.
  • Determinism is the discipline: own your RNG, derive stable IDs by hashing content, and step motion in a synchronous loop with no physics engine. Same seed + version = identical bytes.
  • A scenario is a distribution you sample and log. Bias pixel size small, include empty negatives, and deliberately synthesize terrain and clutter — empty-sky-only data is the classic trap Chapter 7 will make you pay for.
  • Own your assets and run headless. Permissive sources plus a provenance sidecar let the Chapter 5 gate approve every sequence; judge the generator by transfer to real data, not by how good the pictures look.
  • Blockout, then set dressing. A low-strength generative pass can repaint the engine's surface statistics toward realism — but only with the labeled object pixels composited back untouched, a content-hashed seed, a new derived sequence identity, and the model's own license in the sidecar. The generative model dresses the set; it never rebuilds it.
